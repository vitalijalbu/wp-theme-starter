## 1. SCOPO: RIUSABILITÀ SU PIÙ PROGETTI

### ✅ Punti forti
- **Design token system centralizzato** — `theme.json` + `@theme` in CSS sincronizzati via `@roots/vite-plugin`. Cambi il brand in un posto solo.
- **Customizer completo** — 19 impostazioni (social, CTA, footer, newsletter, spedizione gratis, mappa, announcement bar). Zero hardcoding necessario se usato bene.
- **Acorn Service Provider** — `ThemeServiceProvider.php` permette di iniettare dipendenze per progetto senza toccare il core.
- **Blade composable** — sections + partials ben separati, facile da sovra-scrivere per progetto.

### ❌ Problemi di riusabilità

| Problema | File | Impatto |
|---|---|---|
| Footer linka a `/chi-siamo`, `/blog` hardcoded invece del menu | `sections/footer.blade.php` | Alto — da rifare ogni progetto |
| Pattern `contact-section.php` e `cta-banner.php` con tel/email hardcoded | `patterns/*.php` | Alto — cambiarli a mano ogni deploy |
| Font Google (`Poppins`) hardcoded in `app/setup.php` senza filtro | `app/setup.php` | Medio — niente `apply_filters()` |

| CPT slugs (`/portfolio/`, `/team/`, `/faq/`) non filtrabili | `app/post-types.php` | Basso |

**Fix consigliato:** Aggiungi `apply_filters('theme_font_url', $url)` e usa `__('contatti', 'sage')` per i fallback testuali.

## 4. SEO

### ✅ Implementato (fallback nativo)
Il tema rileva Yoast / Rank Math / AIOSEO e sopprime il proprio output:
```php
// setup.php
$seo_plugins = ['wpseo_head', 'rank_math/head', 'aioseo_head'];
if (!array_filter($seo_plugins, fn($h) => has_action($h))) {
    // Stampa JSON-LD + OG tags nativi
}
```

**JSON-LD supportati:** `Article`, `WebPage`, `Person`
**OG tags:** `og:title`, `og:description`, `og:image`, `og:type`, `og:url`
**Twitter Card:** `summary_large_image`

### ❌ Problemi SEO

| Problema | Impatto |
|---|---|
| JSON-LD solo per Article/Page — manca `Product` schema | Alto per e-commerce |
| Nessun `BreadcrumbList` JSON-LD | Medio — i breadcrumb sono renderizzati ma non strutturati |
| `og:image` fallback usa first attachment, non `_thumbnail_id` | Medio — potrebbe selezionare immagine sbagliata |
| Nessun `hreflang` per siti multilingua (Polylang/WPML) | Medio se riusato internazionalmente |
| Canonical URL non gestita nativamente | Alto senza plugin SEO — necessario per archivi WC con filtri |
| Sitemap XML non generata | Alto — richiede plugin o implementazione nativa |
| `rel="nofollow"` non gestito sui link footer social | Basso |

**Fix critico:** Aggiungi `Product` schema per WooCommerce:
```php
// In setup.php, aggiungere per is_product()
$schema = [
    '@type' => 'Product',
    'name' => get_the_title(),
    'offers' => ['@type' => 'Offer', 'price' => $product->get_price()],
];
```

## 7. UPGRADE COMPATIBILITÀ WP (incl. v7)

### Stato attuale
- Usa `block_editor_settings_all` ✅ (non deprecato)
- `register_block_type()` da directory ✅ (stabile da WP 5.5)
- `wp_body_open()` ✅
- `wp_is_block_theme()` non usato — il tema è un **tema classico** con blocchi, non FSE

### ❌ Rischi upgrade WP 6.x → 7.x

| Rischio | Perché | Soluzione |
|---|---|---|
| Il tema è **Classic Theme** — WP 7 spinge forte FSE | Senza `templates/` e `parts/` FSE, il tema sarà legacy | Aggiungere template FSE base (`index.html`, `header.html`, `footer.html`) come fallback |
| `woocommerce_content()` sarà deprecato WC 10+ | WC punta tutto sui WC Blocks | Migrare a `woocommerce/product-collection` e Cart Block (già parzialmente fatto) |
| `get_theme_mod()` nel Customizer — Gutenberg 20+ punta al Global Styles | Il Customizer è in LTS mode in WP core | Valutare migrazione a `theme.json` + Block Bindings API |
| Alpine.js 3 + `x-data` inline — possibile conflitto con WP interactivity API | WP 6.5+ ha la Interactivity API nativa | Nessuna migrazione urgente ma tenerla d'occhio |
| Acorn 6 richiede PHP 8.3 — WP 7 potrebbe richiedere PHP 8.4 | Roots.io la aggiornerà ma è un dependency chain | Pin a `roots/acorn: ^6` e testa su PHP 8.4 |

**Raccomandazione principale:** Aggiungere almeno i template FSE base per non essere marcato "theme classico legacy" in WP 7:
```
templates/
├── index.html      (fallback FSE)
└── 404.html
parts/
├── header.html
└── footer.html
```

## 12. EXTRA: OSSERVAZIONI AGGIUNTIVE

### A. Il tema è un "ibrido" — chiarirlo nella doc

Il tema usa sia template Blade classici **sia** blocchi Gutenberg. Questo crea ambiguità: un developer non sa se fare una nuova sezione come pattern/blocco o come template PHP. Serve una **decision tree** in `CLAUDE.md`:
- Sezione ripetuta in più pagine → Pattern Gutenberg
- Layout unico per un CPT → Template Blade
- Componente interattivo con stato → Alpine component in `app.js`

### B. Nessun sistema di testing

Zero test: niente PHPUnit, niente Vitest, niente Playwright. Per un tema enterprise:
- **PHPUnit** per REST endpoints (almeno smoke test)
- **Playwright** per form e checkout flow
- **axe-core** per a11y automatizzato

### C. Dipendenza GSAP senza licenza verificata

GSAP è gratuito per uso "no charge", ma **ScrollTrigger commerciale richiede licenza Club GreenSock** su siti che vendono. Verificare prima di andare in produzione su e-commerce.

### D. Acorn 6 + WordPress VIP

WordPress VIP ha restrizioni su:
- Scrittura su filesystem (Acorn usa cache su disk) → usare Redis Object Cache invece
- Transient per rate limiting → ok su VIP, ma Redis è preferito
- `wp_mail()` diretto → VIP richiede SMTP esterno

### E. Internazionalizzazione (i18n) incompleta

```php
// Alcune stringhe non sono wrapped:
// customizer.php
'description' => 'URL della mappa Google',  // ← non traducibile!

// Dovrebbe essere:
'description' => __('URL della mappa Google', 'sage'),
```

Una ricerca veloce suggerisce ~40-50 stringhe non wrappate in `__()`.

### F. `biome.json` esclude i template Blade dal linting

```json
// biome.json
"ignore": ["**/*.blade.php"]
```

Blade non è JS, ma le parti `<script>` inline non vengono controllate. Valutare di estrarre tutto il JS da Blade nei moduli.
